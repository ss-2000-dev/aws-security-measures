# 動作確認チェックリスト（AWSコンソール版）

**目的**: 3つのセキュリティ対策の実装後、正しく機能しているかを検証  
**操作方法**: すべてAWSコンソール（ブラウザ）で実施  
**前提**: CloudFormation スタックがすべて CREATE_COMPLETE になっていること

---

## ③ CodeCommit ブランチ/リポジトリ削除保護

### 1️⃣ CloudFormation スタックの確認

```
AWSコンソール → CloudFormation → スタック
→ スタック名「codecommit-protection-[リポジトリ名]」を選択
→ ステータスが「CREATE_COMPLETE」であることを確認
→「リソース」タブ → IAM ポリシーが2個作成されていることを確認
```

- [ ] ステータスが CREATE_COMPLETE

---

### 2️⃣ IAM ポリシーがグループにアタッチされているか確認

```
AWSコンソール → IAM → ユーザーグループ
→ 開発者グループ（例: Developers）を選択
→「許可」タブ
→「DenyCodeCommitDelete-[リポジトリ名]」が表示されていることを確認
→「DenyForcePush-[リポジトリ名]」も表示されていることを確認
```

- [ ] DenyCodeCommitDelete ポリシーが表示されている
- [ ] DenyForcePush ポリシーが表示されている

---

### 3️⃣ Approval Rule Template が設定されているか確認

```
AWSコンソール → CodeCommit
→ 左メニュー「承認ルールテンプレート」
→「protect-main-branch」（設定した名前）が表示されていることを確認
→ テンプレートをクリック → 詳細確認:
   - 必要な承認数: 1
   - ブランチフィルター: refs/heads/main
→「リポジトリへの関連付け」タブ → 対象リポジトリが表示されていることを確認
```

- [ ] Approval Rule Template が存在する
- [ ] 対象リポジトリに関連付けられている

---

### 4️⃣ ポリシーの内容を目視確認（IAM ポリシーシミュレーター）

```
AWSコンソール → IAM → ポリシー
→「DenyCodeCommitDelete-[リポジトリ名]」を検索してクリック
→「JSON」タブ → 以下の内容があることを確認:
   Effect: "Deny"
   Action: "codecommit:DeleteRepository"
```

期待される JSON（一部）:
```json
{
  "Effect": "Deny",
  "Action": "codecommit:DeleteRepository",
  "Resource": "arn:aws:codecommit:ap-northeast-1:アカウントID:リポジトリ名"
}
```

- [ ] JSON に Deny の DeleteRepository が含まれている

---

### 5️⃣ 実際の削除保護テスト（コンソールから）

**開発者用 IAM ユーザーでサインインしてテスト（管理者ではテストにならない）**

```
1. 開発者用 IAM ユーザーでコンソールにログイン
2. AWSコンソール → CodeCommit → リポジトリ一覧
3. 対象リポジトリを選択 → 右上「設定」→「リポジトリを削除」を試みる
4. 削除ダイアログが出た場合はリポジトリ名を入力して「削除」を試みる
```

期待される結果:
- エラーメッセージ「このリポジトリを削除する権限がありません」が表示されること

- [ ] 削除が拒否されエラーが表示された

---

### 概要テーブル: CodeCommit

| 項目 | 確認場所（コンソール） | 期待結果 | 結果 |
|------|-------------------|--------|-----|
| CF スタック | CloudFormation → スタック | CREATE_COMPLETE | ☐ |
| IAM ポリシーのアタッチ | IAM → ユーザーグループ → 許可タブ | 2個のポリシー表示 | ☐ |
| Approval Rule | CodeCommit → 承認ルールテンプレート | テンプレート存在 | ☐ |
| ポリシー内容 | IAM → ポリシー → JSON タブ | Deny DeleteRepository | ☐ |
| 削除テスト | 開発者ユーザーで削除試行 | エラーで拒否 | ☐ |

## ②S3 オブジェクトロック

### 1️⃣ CloudFormation スタックの確認

```
AWSコンソール → CloudFormation → スタック
→ 「s3-lock-」で始まるスタックを確認（15個）
→ すべてのステータスが「CREATE_COMPLETE」であることを確認
```

- [ ] 15個のスタックすべてが CREATE_COMPLETE

---

### 2️⃣ 新バケットのオブジェクトロック設定確認

```
AWSコンソール → S3 → バケット一覧
→ 新しく作成したバケット（例: my-bucket-secured）をクリック
→「プロパティ」タブ
→「オブジェクトロック」セクションを確認:
   オブジェクトロック: 有効
   デフォルトの保持設定:
     モード: GOVERNANCE
     保持期間: 30日
```

- [ ] オブジェクトロックが「有効」と表示されている
- [ ] デフォルト保持モードが GOVERNANCE
- [ ] 保持期間が設定されている（例: 30日）

---

### 3️⃣ アップロードしたオブジェクトにロックが適用されているか確認

```
テスト用オブジェクトのアップロード:
AWSコンソール → S3 → 新バケット
→「アップロード」→ テスト用テキストファイルをドラッグ＆ドロップ
→「アップロード」

ロック情報の確認:
→ アップロードしたオブジェクトをクリック
→「プロパティ」タブ → 下部の「オブジェクトロック」セクションを確認
→ 保持モードと保持期限日が自動で設定されていることを確認
```

期待される表示:
```
オブジェクトロック保持
  保持モード: GOVERNANCE
  保持期限: 2026-07-22T00:00:00Z（今日から30日後）
```

- [ ] 保持モードが表示されている
- [ ] 保持期限日が正しく（本日 + 設定日数）設定されている

---

### 4️⃣ 削除保護テスト（コンソールから）

```
AWSコンソール → S3 → 新バケット
→ アップロードしたテストオブジェクトにチェックを入れる
→「削除」ボタンをクリック
→ 削除確認ダイアログで「完全に削除」と入力して「削除」を試みる
```

期待される結果:
- エラーメッセージが表示され、オブジェクトが削除されないこと
- または削除リクエストが通っても「オブジェクト一覧」から消えていないこと（バージョニングにより復元可能状態で残存）

- [ ] 削除が失敗またはオブジェクトが残存している

---

### 5️⃣ バケットのバージョニング確認

オブジェクトロックにはバージョニングが必須です。

```
AWSコンソール → S3 → バケット → プロパティタブ
→「バケットのバージョニング」セクション
→「有効」になっていることを確認
```

- [ ] バージョニングが「有効」になっている

---

### 6️⃣ 既存バケットからのオブジェクト移行確認

```
移行元の既存バケット:
AWSコンソール → S3 → 既存バケット
→ オブジェクト一覧 → 件数を確認（例: 1,500件）

移行先の新バケット:
AWSコンソール → S3 → 新バケット（-secured）
→ オブジェクト一覧 → 件数が移行元と同数であることを確認（例: 1,500件）
```

- [ ] 移行元と移行先のオブジェクト数が一致している

---

### 概要テーブル: S3

| 項目 | 確認場所（コンソール） | 期待結果 | 結果 |
|------|-------------------|--------|-----|
| CF スタック（15個） | CloudFormation → スタック | 全 CREATE_COMPLETE | ☐ |
| オブジェクトロック設定 | S3 → バケット → プロパティ | 有効 + GOVERNANCE + 30日 | ☐ |
| オブジェクトへの適用 | S3 → オブジェクト → プロパティ | 保持モード・期限あり | ☐ |
| 削除保護テスト | S3 → 削除試行 | 削除失敗・オブジェクト残存 | ☐ |
| バージョニング | S3 → バケット → プロパティ | 有効 | ☐ |
| オブジェクト移行 | 移行元・移行先の件数比較 | 件数一致 | ☐ |

## ①IAM Identity Center セッション時間制限

### 1️⃣ IDC が有効化されているか確認

```
AWSコンソール → IAM Identity Center
→ ダッシュボードが表示されることを確認（エラーでないこと）
→「AWSアクセスポータルのURL」が表示されていることを確認
→「設定」（左メニュー）→「インスタンスの詳細」→ インスタンス ARN がコピーできることを確認
```

- [ ] IDC ダッシュボードが表示されている
- [ ] アクセスポータル URL が発行されている

---

### 2️⃣ CloudFormation スタックの確認

```
AWSコンソール → CloudFormation → スタック
→「idc-permission-sets」スタックを選択
→ ステータスが「CREATE_COMPLETE」であることを確認
→「リソース」タブ → PermissionSet が作成されていることを確認
```

- [ ] ステータスが CREATE_COMPLETE
- [ ] PermissionSet リソースが存在する

---

### 3️⃣ PermissionSet のセッション時間を確認

```
AWSコンソール → IAM Identity Center
→ 左メニュー「許可セット（Permission sets）」
→「Developer」をクリック
→「一般設定」セクション → 「セッション時間」を確認
```

期待される表示:
```
セッション時間: 15 分
```

- [ ] セッション時間が「15分」と表示されている

---

### 4️⃣ PermissionSet にポリシーが設定されているか確認

```
AWSコンソール → IAM Identity Center → 許可セット
→「Developer」をクリック
→「ポリシーと許可の境界」タブ
→ インラインポリシーまたはマネージドポリシーが表示されることを確認
```

- [ ] ポリシーが1個以上表示されている

---

### 5️⃣ ユーザーが PermissionSet に割り当てられているか確認

```
AWSコンソール → IAM Identity Center → AWSアカウント
→ 対象のAWSアカウントをクリック
→「ユーザーとグループ」タブ
→ Developers グループ（または個別ユーザー）が表示されていることを確認
→「割り当てられた許可セット」列に「Developer」が表示されていることを確認
```

- [ ] ユーザー/グループが表示されている
- [ ] 割り当てられた許可セットに「Developer」がある

---

### 6️⃣ 実際のログインでセッション時間を確認（手動テスト：最重要）

```
テスト手順:
1. AWSアクセスポータルのURLをブラウザ（シークレットウィンドウ推奨）で開く
2. IDC ユーザーのメールアドレスとパスワードでログイン
3. AWSアカウントの一覧が表示されることを確認
4. 「マネジメントコンソール」リンクをクリックしてコンソールにアクセス
5. コンソール右上のユーザー名（ロール名）を確認
6. 15分間、コンソールで何も操作せずに待機
7. 「セッションの有効期限が切れました」などのメッセージが表示されることを確認
8. 再度ログインを求められることを確認
```

- [ ] アクセスポータルからコンソールにログインできた
- [ ] 15分後に自動でセッションが切れた（ログアウトまたは再認証要求）

---

### 7️⃣ IAM グループのメンバーが IDC グループに移行しているか確認

```
移行前（IAM）のメンバー確認:
AWSコンソール → IAM → ユーザーグループ → Developers
→「ユーザー」タブ → メンバー数を確認（例: 10人）

移行後（IDC）のメンバー確認:
AWSコンソール → IAM Identity Center → グループ
→ Developers グループをクリック
→「グループメンバー」タブ → メンバー数が一致することを確認
```

- [ ] IAM グループと IDC グループのメンバー数が一致している

---

## 📋 最終確認チェック（コンソールまとめ）

### すべての確認を1画面で行う順序

```
1. CloudFormation → スタック → フィルタ「CREATE_COMPLETE」で全スタック確認
2. IAM → ユーザーグループ → Developers → 許可タブ → CodeCommit ポリシー確認
3. CodeCommit → 承認ルールテンプレート → 設定確認
4. S3 → ロック対応バケット → プロパティ → オブジェクトロック確認
5. IAM Identity Center → 許可セット → Developer → セッション15分確認
6. 実際にアクセスポータルでログイン → 15分テスト
```

## トラブルシューティング（コンソール上）

| 症状 | 確認場所 | 対処法 |
|------|---------|-------|
| CF スタックが ROLLBACK | CloudFormation → スタック → イベントタブ | エラーメッセージを確認して原因特定 |
| ポリシーがグループに見当たらない | IAM → ポリシー で名前検索 | スタックが正常完了しているか再確認 |
| S3 のオブジェクトロック欄が空 | バケット → プロパティ | バケット作成時に ObjectLock を有効化したか確認 |
| IDC にログインできない | IDC → ユーザー → ユーザー詳細 | 招待メールを確認・パスワード設定済みか確認 |
| セッションがすぐ切れない | IDC → 許可セット → セッション時間 | 設定が保存されているか・再プロビジョニングが必要か確認 |
