# 開発者への影響分析

**対象**: 3つのセキュリティ対策導入時の開発者への実際の影響  
**重要**: ビジネスロジックには影響なし。ただし作業フロー / UX に影響あり

---

## ①IAM Identity Center：セッション時間15分制限

### 🔴 高い影響度

#### 実際の開発者体験

```
現在（IAM ユーザー）
┌──────────────────────────────────┐
│ ログイン                          │
│ ↓                                │
│ コンソール操作（ずっと有効）      │
│ ↓                                │
│ 複数タスク同時実行                │
│ ↓                                │
│ ログアウト時のみ終了              │
└──────────────────────────────────┘
```

```
移行後（IDC + 15分セッション）
┌──────────────────────────────────┐
│ ログイン                          │
│ ↓                                │
│ EC2 確認（5分）                    │
│ ↓ ━━━━━━━━━━━━━━━━━━━━━━━━━━━━ │
│ セッション期限切れ（15分後）      │
│ ↓ ⚠️ ⚠️ ⚠️                      │
│ 「再度認証してください」エラー    │
│ ↓                                │
│ 再度ログイン（手動作業 2-3分）    │
│ ↓                                │
│ S3 確認（3分）                    │
│ ↓ ━━━━━━━━━━━━━━━━━━━━━━━━━━━━ │
│ セッション期限切れ                │
│ ↓                                │
│ 再度ログイン（2-3分）             │
└──────────────────────────────────┘

⏰ 追加の再ログイン時間: 4-6分/日
```

### 具体的な影響

| 操作 | 現在 | 移行後 | 追加時間 |
|------|------|-------|--------|
| EC2 インスタンス確認 | ログイン後すぐ | ログイン + 操作 | 0分 |
| 同じコンソール内で複数操作 | 15分ずっと有効 | 15分で期限切れ | +2-3分（再ログイン） |
| 昼休みから帰社後 | そのまま利用可 | 再度ログイン必須 | +2-3分 |
| CloudFormation スタック作成監視 | スタック完成まで待機可 | 期限切れになる（ ⚠️ ） | +10-20分（監視不可） |

### 🔴 リスク: CloudFormation 長時間実行時の監視不可

```
CF スタック作成: 20分かかる場合
├─ 最初の15分: 開発者がコンソール監視 → セッション期限切れ
├─ 次の5分: 開発者はログインし直し（別のセッション）
└─ ⚠️ 最初のセッションでのエラー通知が見えない可能性
```

**対策**: CLI ベースのスタック監視に変更
```bash
# より適切な方法
aws cloudformation wait stack-create-complete \
  --stack-name my-stack

# または
watch 'aws cloudformation describe-stacks \
  --stack-name my-stack \
  --query "Stacks[0].StackStatus"'
```

### 💡 開発者側での対応策

1. **長時間作業の前にログイン**
   - 朝ログイン → 午前の作業（15分以内に完了）
   - 昼休み → ログアウト + ランチ
   - 再度ログイン → 午後の作業

2. **CLI/SDK への切り替え**
   - コンソールでの長時間監視が不可 → AWS CLI 推奨
   - API 呼び出しは TokenExpiration でハンドリング可能

3. **多くのタスクをバッチ化**
   - 複数のリソース確認 → スクリプト化して一度に実行

### 📊 適応期間

| 期間 | 適応状況 |
|------|--------|
| 1-2週間 | ❌ かなり不便（忘れて何度も再ログイン） |
| 2-4週間 | ⚠️ 徐々に慣れていく |
| 1ヶ月以降 | ✅ 習慣化（再ログインが当たり前に） |

## ②S3 オブジェクトロック：既存オブジェクト削除不可

### 🟡 中程度の影響

#### 実際の開発者体験（開発環境）

```
現在（開発環境 S3 バケット）
┌─────────────────────────────────┐
│ テスト用データをアップロード      │
│ ↓                              │
│ テスト実行（バグ発見）            │
│ ↓                              │
│ 「あ、このデータは不要だ」        │
│ ↓                              │
│ s3 rm s3://bucket/test-data.txt │
│ ↓                              │
│ 修正版を再度アップロード         │
│ ↓                              │
│ テスト再実行 → OK                │
└─────────────────────────────────┘
⏱️ 作業時間: 2-3分
```

```
移行後（S3 オブジェクトロック有効）
┌─────────────────────────────────┐
│ テスト用データをアップロード      │
│ ↓                              │
│ テスト実行（バグ発見）            │
│ ↓                              │
│ 「あ、このデータは不要だ」        │
│ ↓                              │
│ s3 rm s3://bucket/test-data.txt │
│ ↓ ❌                            │
│ エラー: オブジェクト削除不可      │
│ (Retention: 14日)               │
│ ↓                              │
│ 代案1: タグで「削除予定」マーク   │
│       → 14日後に自動削除        │
│ ↓ OR                            │
│ 代案2: 異なる名前で再度アップロード│
│       → test-data-v2.txt        │
│ ↓                              │
│ テスト再実行 → OK                │
└─────────────────────────────────┘
⏱️ 作業時間: 3-5分 + 管理作業
```

### 具体的な影響

#### ケース1: 開発環境での繰り返しテスト

```
通常のテストサイクル:
1. データアップロード → テスト → 削除 → 修正 → 再度テスト

ロック後:
1. データアップロード → テスト → ❌ 削除不可
2. 工夫1: 別ファイル名で再度アップロード
   test-data.txt, test-data-v2.txt, test-data-v3.txt...
   → ストレージ使用量増加（消去できないため）

3. 工夫2: オブジェクトタグで「削除予定」マーク
   開発者が手動で削除を依頼するワークフロー
   → 管理作業が増える
```

#### ケース2: ストレージコスト

```
開発環境の S3 バケット容量変化:

現在:
月間 50GB アップロード
→ 毎日削除
→ 常時 5GB 使用

移行後:
月間 50GB アップロード
→ 削除不可（14日保持）
→ 常時 25GB 使用 ⚠️（5倍に）

追加コスト: $1/月程度（$0.023/GB）
```

### Governance vs Compliance モード

ユーザーの選択基準:

| モード | 削除できるユーザー | 開発者への制約 | 推奨 |
|------|---------|---------|-----|
| **Governance** | AWS Account Owner（root権限） + 特定IAM | 通常ユーザーは削除不可 | ⭐⭐⭐ 推奨 |
| **Compliance** | 誰も削除できない（root含む） | **完全に不可** | リスク高（本番のみ） |

### 💡 開発者側での対応策

**戦略1: 開発バケットと本番バケットを分離**
```
開発環境: ロックなし（現在のまま）
本番環境: ロック有効（GOVERNANCE, 30日）

→ 開発は自由、本番は保護
```

**戦略2: ロック期間を短く設定**
```
設定例:
- 開発環境: 3日（修正テストが完了する期間）
- 本番環境: 30日（規制要件）

→ 開発ストレスを最小化
```

**戦略3: 自動化されたテストデータ削除**
```bash
# テスト終了後、タグで標識
aws s3api put-object-tagging \
  --bucket dev-bucket \
  --key test-data.txt \
  --tagging 'TagSet=[{Key=DeleteAfter,Value=2026-06-23}]'

# Lambda で毎日チェック
# → 期限切れのオブジェクトを削除要求
```

### 🟡 実際の開発フロー変化

```
回数　　　　 現在               移行後             追加作業
1-10回目    削除してテスト  →  削除不可・タグ付け  手動タグ付け（毎回1分）
11-20回目   削除してテスト  →  別ファイル名利用   ファイル名を工夫（頭脳労働）
21回目以上  ストレージ圧迫      不要ファイルが蓄積  月1回のクリーンアップ

⏰ 1開発者当たり追加時間: **5-10分/週**
```

## ③CodeCommit ブランチ削除保護

### 🟢 低い影響

#### 実際の開発者体験

```
現在
┌────────────────────────────┐
│ feature-branch で開発        │
│ ↓                          │
│ PR レビュー → 承認         │
│ ↓                          │
│ Merge to main              │
│ ↓                          │
│ git push origin --delete   │
│   feature-branch           │
│ ↓                          │
│ ✅ 削除完了                │
└────────────────────────────┘
⏱️ 追加時間: 0秒（削除は自動処理）
```

```
移行後
┌────────────────────────────┐
│ feature-branch で開発        │
│ ↓                          │
│ PR レビュー → 承認         │
│ ↓                          │
│ Merge to main              │
│ ↓                          │
│ git push origin --delete   │
│   feature-branch           │
│ ↓ ✅                       │
│ 削除完了（feature は自由）  │
│ ↓ ❌                       │
│ main branch の削除を試みる  │
│ Error: 削除不可             │
└────────────────────────────┘
⏱️ 追加時間: 0秒（main 削除は不可能）
```

### 具体的な影響

| 操作 | 現在 | 移行後 | 影響 |
|------|------|-------|-----|
| feature-branch 削除 | ✅ 可能 | ✅ 可能 | ❌ なし |
| develop ブランチ削除 | ✅ 可能 | ✅ 可能（保護なし場合） | ❌ なし |
| **main ブランチ削除** | ✅ **危険** | ❌ **不可** | ✅ **安全になった** |
| Force push to main | ✅ 可能 | ⚠️ 制限可能 | ⚠️ ブランチ保護ルール次第 |

### 🟢 むしろ開発者にとって安全

```
よくあるミス:
「git push origin :main」で誤ってmainを削除
→ リポジトリ全体が壊れる ❌

移行後:
同じコマンドを実行 → 「Denied: main is protected」
→ リポジトリが保護されている ✅
```

### 💡 開発者への説明ポイント

> **「main ブランチと develop ブランチは削除できなくなります。」**
> 
> これは誤った削除を防ぐためです。
> - feature-branch や release-branch は普通に削除できます
> - main/develop の削除が必要な場合は、セキュリティチームに申請してください

## 📊 総合的な影響マトリックス

| 対策 | 影響度 | 主な影響範囲 | 開発者への負担 | 適応期間 |
|-----|-------|--------|----------|--------|
| IAM Identity Center | 🔴 高 | 日々のログイン体験 | 高（再ログイン頻繁） | 1-4週間 |
| S3 オブジェクトロック | 🟡 中 | テスト/開発フロー | 中（削除できない） | 1-2週間 |
| CodeCommit 保護 | 🟢 低 | ブランチ管理 | 低（main 削除不可） | 数日 |

## 🎯 開発者向け実施ガイド

### アナウンスのタイミング

```
理想的なスケジュール:

Week 1: 告知 + 事前準備
  - セキュリティ対策実装の予定を通知
  - 各対策の影響を説明
  - Q&A セッション

Week 2: CodeCommit + S3 実装
  - 最初に影響の低い対策から実装
  - 開発者がすぐに対応できる

Week 3-4: IAM Identity Center 準備
  - IDC インスタンス構築
  - テストアカウントで動作確認

Week 5: IAM Identity Center 本番移行
  - 段階的なユーザー移行
  - ヘルプデスク体制を整備
```

### オンボーディング資料

```
開発者向け FAQ:

Q1: なぜセッション時間が15分に?
A: セキュリティ監査の要件です。万が一ノートパソコン盗難時に、
   自動的にセッションが無効化され、被害を最小化できます。

Q2: 長い作業の場合はどうする?
A: CLI/SDK を利用するか、途中で再度ログインしてください。

Q3: オブジェクトロックで削除できない?
A: 開発バケットと本番バケットを分離しています。
   開発環境ではロックなし（現在のまま）です。

Q4: main ブランチを削除したい場合は?
A: セキュリティチームに申請してください（通常は不要）。
```

## 💡 実装後の運用トips

### 開発者からのよくあるサポート依頼

| 依頼 | 理由 | 対応 |
|------|------|-----|
| 「セッションが切れた」 | 15分経過 | 再度ログイン（仕様） |
| 「オブジェクトが削除できない」 | ロック中 | 削除予定日を確認 |
| 「main が削除できない」 | 保護中 | feature ブランチを利用 |
| 「IDC ログインができない」 | アカウント未作成 | IDC グループに追加 |

### ダッシュボード/モニタリング

```
推奨: 以下を週1回確認
- Approve Rule 違反（force push 試行）
- S3 オブジェクトロック期限切れ（残り1週間）
- IAM Identity Center セッション期限切れエラー（CloudWatch ログ）
```

---

**まとめ**: 
- ✅ CodeCommit は影響最小限（むしろ安全性向上）
- ⚠️ S3 はストレージ戦略を見直す必要
- ⛔ IAM IDC は適応期間が必要（事前告知重要）
