# 実装優先順位 & カフェ（1時間）準備ガイド

**目的**: 明日朝1時間のカフェ学習を効率化  
**操作方法**: すべてAWSコンソール（ブラウザ）で実施  
**実装開始時期**: カフェ後、その週のうちから段階的に開始可能

---

## 🎯 実装優先順位の理由

### 推奨実装順序

```
1️⃣ CodeCommit (1日)     → 最も低リスク、すぐ完了 ✅
2️⃣ S3 (1日)             → 中程度難易度、計画的に進行可能
3️⃣ IAM Identity Center  → 最も複雑、テスト期間が必要 ⏳
   (4-5日)
```

### なぜこの順序か？

```
CodeCommit を最初にする理由:
① リスクが最も低い（削除保護が追加されるだけ）
② CloudFormation デプロイ後すぐに動作確認できる
③ 1日で完了するため達成感を得やすい

IAM Identity Center を最後にする理由:
① IDC 有効化がコンソール手動作業
② ユーザー移行に 1-2週間の並行運用期間が必要
③ 開発者への影響が最も大きく、事前告知が必要
```

### Step 2: CodeCommit の学習（8分）

**理解すべき内容**

```
□ 「DeleteRepository を Deny する IAM ポリシー」で削除を防ぐ仕組み
□ Approval Rule Template → PR に承認が必要になる仕組み
□ Force push 防止 → 特定ブランチへの強制上書きを防ぐ
```

**コンソールでの操作の流れを頭に入れる**

```
1. CloudFormation にYAMLをアップロード → ポリシーが自動作成される
2. コンソール（IAM）でポリシーを開発者グループにアタッチ
3. コンソール（CodeCommit）でApproval Rule Template を作成・リポジトリ関連付け
4. 開発者ユーザーでログインしてリポジトリ削除が失敗することを確認
```

**確認**: 動作確認チェックリスト.md の CodeCommit セクション（5分）

---

### Step 3: S3 オブジェクトロックの学習（12分）

**理解すべき内容**

```
□ オブジェクトロック = 削除・上書きを一定期間不可にする
□ GOVERNANCE モード = 管理者は特定権限で解除可能（こちらを推奨）
□ COMPLIANCE モード = 誰も解除不可（本番重要データのみ）
□ 既存バケットへの後付けは不可 → 新バケット作成が必須
□ CloudFormation で「ObjectLockEnabled: true」はバケット作成時のみ有効
```

**コンソールでの操作の流れを頭に入れる**

```
1. CloudFormation にYAMLをアップロード → 新バケットが自動作成される
   （15個のバケット分 → 15スタック）
2. コンソール（S3）で既存バケットのオブジェクトを新バケットへコピー
   （S3 Batch Operations または手動）
3. コンソール（S3）で新バケットのプロパティからロック設定を確認
4. テストオブジェクトをアップロードして削除を試みて拒否されることを確認
```

**開発者への影響を理解**: 開発者への影響分析.md の S3 セクション（3分）

```
重要:
- 開発環境と本番環境でバケットを分ける場合はロック不要なものも選別できる
- GOVERNANCE モードなら管理者が緊急時に解除可能
```

**確認**: 動作確認チェックリスト.md の S3 セクション（4分）

---

### Step 4: IAM Identity Center の学習（15分）

**理解すべき内容**

```
□ IDC = IAM ユーザーの代替となる統合認証基盤
□ PermissionSet = IAM ロールに相当する概念
□ セッション時間: CloudFormation の SessionDuration: PT15M で設定
□ IDC は組織全体で1回だけ有効化（コンソールのみ）

移行の流れ:
  現在の IAM グループ1個 → IDC グループ1個
  現在の IAM カスタムポリシー5個 → IDC PermissionSet 5個

並行運用期間:
  移行中は IAM ユーザー と IDC ユーザーが共存する（1-2週間）
```

**コンソールでの操作の流れを頭に入れる**

```
1. コンソール（IAM Identity Center）で IDC を有効化
2. コンソールで「インスタンス ARN」をコピー
3. CloudFormation にYAMLをアップロード（ARNをパラメータに入力）→ PermissionSet 作成
4. コンソール（IDC）でユーザー・グループを作成
5. コンソール（IDC）でAWSアカウントに PermissionSet を割り当て
6. アクセスポータルURL でテストユーザーがログインできることを確認
7. 15分待機 → セッションが切れることを確認
```

**核となるYAML の要点**

```yaml
SessionDuration: PT15M  # ← ISO 8601形式。P=期間、T=時刻区切り、15M=15分
                        # PT1H=1時間 / PT30M=30分 も同様に記述
```

**開発者への影響**: 開発者への影響分析.md の IAM IDC セクション（3分）

```
最大の懸念事項:
- コンソール操作中に15分経過すると自動ログアウト
- 長時間のCloudFormation監視はコンソールで難しくなる
→ 対策: スタックのイベントタブはときどき手動でリロードして確認
```

**確認**: 動作確認チェックリスト.md の IAM IDC セクション（3分）

### Step 5: CFテンプレートの確認と保存（10分）

**3つのテンプレートファイルを手元に準備**

```
CloudFormation_テンプレート集.md を開いて、
以下の3つのYAML をそれぞれテキストエディタにコピーして保存:

1. codecommit-protection.yaml
   → CodeCommit セクションの YAML 全体
   
2. s3-object-lock.yaml
   → S3 セクションの YAML 全体
   
3. idc-permission-sets.yaml
   → IAM Identity Center セクションの YAML 全体

保存場所の例:
  デスクトップ/aws-templates/codecommit-protection.yaml
  デスクトップ/aws-templates/s3-object-lock.yaml
  デスクトップ/aws-templates/idc-permission-sets.yaml
```

**各テンプレートで自分の環境に合わせて修正が必要な箇所**

```
codecommit-protection.yaml:
  → Default: my-secure-repo ← 実際のリポジトリ名に変更（デプロイ時にパラメータで指定）

s3-object-lock.yaml:
  → BucketName ← 新バケット名をデプロイ時に指定
  → RetentionDays: 30 ← 保持期間（日数）を確認して変更

idc-permission-sets.yaml:
  → SSOInstanceArn ← デプロイ時にコンソールからコピーした ARN を入力
  → InlinePolicy の中身 ← 既存の IAM カスタムポリシー 5個の内容を移植
```

### Step 6: リスク確認（5分）

**実装前の最終確認**

```
CodeCommit:
  ❓ 現在使用中のリポジトリ名をすべてリストアップできているか？
     → YES なら実装 OK（10個分スタックを作成するだけ）

S3 オブジェクトロック:
  ❓ 本番バケットと開発バケットを区別できているか？
     → どのバケットにロックをかけるか選別する（全15個？本番のみ？）
  ❓ オブジェクトコピー中のダウンタイムをゼロにする必要があるか？
     → あるなら S3 レプリケーション（非同期コピー）を検討

IAM Identity Center:
  ❓ 既存 IAM カスタムポリシー 5個の JSON 内容はメモしてあるか？
     → IDC PermissionSet に移植するので事前確認が必要
  ❓ 社内に IAM Identity Center の経験者はいるか？
     → 初回導入は注意点が多いので確認推奨
```

## 🎯 カフェ帰宅後のアクション

### Day 1-2（CodeCommit）

```
AWSコンソールでの作業:
1. CloudFormation → codecommit-protection.yaml をアップロード
   パラメータ: RepositoryName = [実際のリポジトリ名1]
   → CREATE_COMPLETE を確認

2. IAM → ユーザーグループ → Developers
   → 「DenyCodeCommitDelete-[リポジトリ名]」をアタッチ
   → 「DenyForcePush-[リポジトリ名]」をアタッチ

3. CodeCommit → 承認ルールテンプレート → 作成
   → 対象リポジトリに関連付け

4. Step 1-3 をリポジトリ10個分くり返す

5. 動作確認チェックリストで全項目確認
```

### Day 3-5（S3）

```
AWSコンソールでの作業:
1. CloudFormation → s3-object-lock.yaml をアップロード
   パラメータ: BucketName = [既存バケット名]-secured
   → CREATE_COMPLETE を確認
   → バケット15個分 × 15スタック 作成

2. S3 → 既存バケット → オブジェクトコピー → 新バケットへ
   （S3 Batch Operations または手動）

3. 動作確認チェックリストで全項目確認
```

### Day 6-20（IAM Identity Center）

```
AWSコンソールでの作業:
1. IAM Identity Center を有効化
   → インスタンス ARN をメモ

2. CloudFormation → idc-permission-sets.yaml をアップロード
   パラメータ: SSOInstanceArn = [メモした ARN]
   → CREATE_COMPLETE を確認

3. IDC コンソール:
   → グループ作成（Developers）
   → ユーザー作成（IAM ユーザー数分）
   → AWSアカウントに PermissionSet を割り当て

4. テストユーザーでアクセスポータルからログイン確認
   → 15分でセッション切れを確認

5. 本番ユーザーを段階的に IDC へ移行（1-2週間）
```
