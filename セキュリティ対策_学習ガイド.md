## 🔄 ①IAM Identity Center への移行

### 実装フロー
```
現在の状態          →          移行プロセス              →          完成状態
┌─────────────┐              ┌──────────────────┐          ┌──────────────┐
│ IAM ユーザー │              │1. IDC有効化       │          │IDC ユーザー  │
│ IAM グループ │  移行準備 →  │2. グループ作成    │  制御 → │ IDC グループ │
│ 　　　　　  │              │3. ポリシー同期   │    ←    │ ポリシーセット│
│ IAM ポリシー│              │4. テスト確認      │          │セッション    │
└─────────────┘              │5. カットオーバー  │          │15分制限 ✅   │
                             └──────────────────┘          └──────────────┘
```

### CloudFormation 対応状況
| 工程 | CloudFormation 対応 | 手動作業 |
|------|------------|------|
| IDC 有効化 | ❌ | AWS Console のみ |
| ユーザー/グループ作成 | ✅ | - |
| ポリシーセット作成 | ✅ AWS::SSO::PermissionSet | - |
| セッション期間設定 | ✅ PermissionSet内で設定 | - |
| IAM→IDC 移行 | 部分的 | 手動でデータ移行 |

### 重要: IAM ユーザーとの並行期間
- IDC ユーザーが全員移行完了まで両方管理
- 移行期間: **1-2週間** が現実的
- **IAM ポリシー5個 → IDC PermissionSet 5個** に変換

### 開発者への影響
```
現在（IAM）: ログイン情報 → ずっと有効
    ↓ 移行後（IDC）
新規（IDC）: ログイン情報 → 15分ごとに再認証
           （コンソール操作のたびに再ログイン可能性）
```

### 工数見積もり
- **準備フェーズ**: 1.5日
  - IDC 構成設計
  - グループ/ポリシーマッピング
  - テスト環境構築
- **実装フェーズ**: 1.5日
  - CloudFormation テンプレート作成
  - テスト実行
- **移行フェーズ**: 1-2日
  - 段階的なユーザー移行
  - トラブルシューティング
- **合計**: **4-5日**（実作業）

## 🔒 ②S3 オブジェクトロック

### オブジェクトロックの動作
```
ロック設定前                  ロック設定後
┌──────────────┐             ┌──────────────┐
│ オブジェクトA │             │ オブジェクトA │
│ 作成済み      │     →      │ [Lock Mode] │
│ 自由に削除可  │             │ Retention   │
└──────────────┘             └──────────────┘
                              ロック期間内は:
                              ✅ 読み取り: OK
                              ❌ 更新: NG
                              ❌ 削除: NG
                              ❌ タグ変更: NG
```

### オブジェクトロック2つのモード
| モード | 用途 | 効果 | 開発者への制約 |
|------|------|------|-------|
| **Governance** | 一般的な保護（推奨） | 削除不可（14日） | 💥  開発時に再テスト不可 |
| **Compliance** | 強制的保護 | 削除完全不可 | 💥💥💥 完全に不可（root含む） |

### 既存オブジェクトへのロック適用
⚠️ **制限事項**
- S3 Select, Batch Operations を使用して後付けロック設定
- ただし、バケット自体に ObjectLock 有効化が必須
- 既存バケットには ObjectLock 追加できない → **新バケット作成 + マイグレーション必要**

### 開発者への実際のデメリット
```
現在: 開発テスト
  ↓
1. オブジェクトアップロード → テスト
2. バグ発見 → オブジェクト削除
3. 修正 → 再アップロード → テスト
4. ✅ 完了

ロック後: 開発テスト
  ↓
1. オブジェクトアップロード → テスト
2. バグ発見 → ❌ オブジェクト削除BLOCKED
3. 代わりにタグで「削除予定」マーク
4. Retention期間終了後に削除
5. ⏰ 遅延が発生
```

### 工数見積もり
- **準備**: 0.5日
  - バケット一覧確認
  - ロック戦略決定（Governance/Compliance）
  - リテンション期間決定
- **実装**: 0.5日
  - 新バケット作成（CF）
  - オブジェクト移行スクリプト
  - 設定確認
- **合計**: **1日**（1つのバケット当たり 2-3時間、15個で1日で完了可能）

### 動作確認での注意点
- Governance モード設定後も **root アカウント + 特定 IAM 権限で削除可能**
- テスト時は意図的に削除を試みて NG になることを確認

## 🛡️ ③CodeCommit ブランチ/リポジトリ削除保護

### 保護の種類
| 保護タイプ | 対象 | 効果 | 実装方法 |
|---------|------|------|-------|
| **Repository 削除保護** | リポジトリ全体 | リポジトリ削除NG | 📝 RepositoryPolicy IAM |
| **Branch 削除保護** | 特定ブランチ（通常main） | ブランチ削除NG | ⚙️ Approval Rule + Policy |
| **Force push 防止** | すべてのブランチ | 強制push NG | ⚙️ Approval Rule |

### CloudFormation での実装
```yaml
# リポジトリ削除保護
- Repository に ResourcePolicy 追加（Deny DeleteRepository）

# Branch 保護（Approval Rule）
- ApprovalRuleTemplate または ApprovalRule 作成
- デフォルトでmain ブランチに適用
```

### 工数見積もり
- **準備**: 0.5日
  - リポジトリ一覧確認
  - 保護対象ブランチ決定（mainのみ? develop も?）
- **実装**: 0.5日
  - CloudFormation テンプレート作成
  - 10個のリポジトリに適用
  - 確認
- **合計**: **1日**

### 開発者への影響
- ✅ **最小限**
- 削除したい場合: セキュリティチーム経由で解除依頼（ワークフロー化推奨）
- Force push は Approval Rule で検出可能
